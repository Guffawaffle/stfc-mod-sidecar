# Producer/Consumer Security Contract

This document is the architecture contract between STFC Community Mod and STFC Mod Sidecar.

The short version: the mod is a producer, the sidecar is a consumer. The mod may emit data through narrow, configured channels, but gameplay and mod hooks must continue even when the sidecar is slow, absent, misconfigured, or overloaded. Privacy and security own the boundary before AX, UI, analytics, or integrations shape it.

## Goals

- Keep the injected mod small, low-latency, and independent from sidecar availability.
- Let the sidecar do expensive parsing, storage, packaging, UI, AX, and integration work on its own schedule.
- Keep the event data plane secret-free and portable.
- Keep auth, egress, and privileged operations out-of-band from event payloads.
- Make local and remote targets use the same target-scoped contract shape without sharing keys across trust zones.

## Core Invariants

1. The mod must not wait on sidecar processing in a gameplay-critical path.
2. The sidecar must not reach into the game process or become a gameplay control channel.
3. Events are data-plane observations, not authorization envelopes.
4. Secrets are auth-plane state and must never appear in JSONL, battle payloads, AX packages, logs, or diagnostic bundles by default.
5. Backpressure is advisory to the producer. If the consumer cannot keep up, the producer drops, samples, degrades, or disables optional export instead of blocking the game.
6. Remote egress is target-scoped and explicit. Local persistence is not consent for remote upload.

## Trust Zones

### Community Mod

The mod observes game/mod state and emits structured data. It owns only the minimum target credentials needed to deliver configured sync/export traffic.

The mod does not own:

- sidecar UI state
- sidecar storage policy
- provider integrations
- provider credential lifecycle
- user-facing export review
- gameplay automation through the sidecar

### Sidecar Core

The sidecar validates, stores, indexes, summarizes, packages, and displays data. It owns durable local storage and any later provider integration workers.

The sidecar does not own:

- game process hooks
- Scopely/game session credentials
- gameplay command execution
- blocking decisions inside mod hot paths

### AX And UI Clients

AX and UI are consumers of sidecar-owned packages and APIs. They do not define the mod contract. If AX or UI needs richer data, sidecar core should expose a safer package first, then UI can render it.

## Data Plane

Allowed producer transports:

- Append-only JSONL for zero-service local fallback and offline replay.
- Localhost HTTP ingest for sidecar-owned storage.
- Future local transports such as named pipes or WebSockets only if they preserve one-way data export and bounded producer cost.

Event payloads may include observations, IDs, timings, decoded battle data, hook status, and diagnostic metadata. Event payloads must not include bearer tokens, cookies, Scopely session headers, provider credentials, local capability tokens, or opaque secret blobs.

Consumers must tolerate:

- missing event families
- unknown additive fields
- invalid JSONL lines
- duplicate events
- out-of-order arrival
- absent sidecar service
- partial batches

## Auth Plane

Every privileged channel gets its own key class. Keys are never shared just because two routes are local.

| Key class | Generated by | Held by | Scope | Persistence | Used for |
| --- | --- | --- | --- | --- | --- |
| Local sync key | Sidecar or managed launcher | Sidecar + configured mod target | Local ingest only | Prefer launch-scoped; dev override allowed | `POST /api/events`, fleet sync ingest |
| Remote sync key | Remote provider or explicit user config | Mod target config | One remote target/profile | Provider policy | Direct mod-to-remote sync target |
| Local capability key | Sidecar core | Trusted local UI/bootstrap client | Local privileged sidecar APIs | Launch-scoped | Admin/settings/privileged local routes |
| Settings key | Sidecar/deployment policy | Trusted settings client | Settings mutation | Deployment policy | Remote-protected settings writes |
| Provider credential | Provider/user via sidecar | Sidecar credential store only | Provider/profile/scopes | OS-backed secret storage | User-approved outbound integrations |
| Game session secret | Game/mod runtime | Mod only | Scopely/game APIs | Mod runtime policy | Game journal/API access |

Local sidecar ingest should use a sidecar-generated sync key when the sidecar launches the service. Remote ingest should use a remote-generated or user-provisioned target key. Neither key should be copied into event payloads, logs, diagnostics, or AX package output.

## Producer Contract

The mod should:

- Treat every export target as optional.
- Use target-scoped config for URL, enabled features, key, and cost controls.
- Use short timeouts for realtime HTTP export.
- Batch when it reduces cost without delaying gameplay-critical work.
- Bound queues and retry budgets.
- Prefer dropping or sampling diagnostic events over delaying the game.
- Keep JSONL retention bounded when local fallback is enabled.
- Emit enough version/session metadata for consumers to correlate data later.
- Redact or omit secret material before logging or emitting.

The mod should not:

- Require sidecar startup for game/mod startup.
- Retry forever on a dead target.
- Block hooks on sidecar parsing or storage.
- Let remote targets inherit local sidecar keys.
- Let sidecar routes become a command channel back into gameplay.

## Consumer Contract

The sidecar should:

- Accept data quickly and defer expensive work.
- Validate envelopes before storage.
- Store raw accepted events for replay/inspection when policy allows.
- Build indexed summaries for AX/UI instead of forcing clients to scan raw payloads.
- Keep battle/default read surfaces separate from runtime/debug read surfaces.
- Reject developer/runtime event ingest in standard mode.
- Expose pressure status and counts to AX before building UI workflows.
- Own local retention, rate limiting, and future egress review.

The sidecar should not:

- Parse inside the mod process.
- Assume localhost means authorized.
- Echo secrets back through API responses.
- Upload local data remotely just because it was stored locally.
- Hide ingestion drops or validation failures from diagnostics.

## Pressure Valves

Producer-side valves:

- feature toggles per target and event family
- runtime trace levels
- bounded in-memory queues
- bounded JSONL replay/retention windows
- short connect/request timeouts
- retry caps with cooldown
- event sampling or summary mode for high-frequency diagnostics
- payload size caps before send

Consumer-side valves:

- maximum ingest body size
- event schema validation
- duplicate detection
- event type filtering and scoped APIs
- summary-first AX packages
- store retention policy
- rate limiting for local ingest and future remote egress
- dead-letter/invalid-line visibility without blocking valid events
- developer-mode gates for runtime diagnostic streams

Backpressure rule: consumer pressure can disable, slow, or reject optional ingest, but it must not force the producer to stall gameplay. The producer records best-effort failure metadata when affordable and then moves on.

## Local Versus Remote

Local sidecar ingest:

- loopback only by default
- sidecar-generated sync key preferred
- local durable storage owned by sidecar
- no remote egress implied
- developer/runtime events require developer mode

Remote sync targets:

- explicit target URL and key
- remote-generated or user-provisioned sync key
- target-specific schema/feature allowlist
- stricter data minimization than local storage
- no automatic reuse of local sidecar sync keys

Local and remote targets can share event schemas only when the target is allowed to receive that event family. They must not share credential material.

## Privacy And Security Priority

Privacy/security decisions are not marketing copy and should not become noisy end-user slogans. They are engineering gates:

- classify data before exposing it
- keep secrets out of portable artifacts
- make egress explicit and reviewable
- prefer redacted summaries for AX/UI
- require opt-in for high-volume or sensitive streams
- add tests before broadening routes or event families

AX and UI should consume already-safe packages. They should not become the first place sensitive data is filtered.

## Allowed And Forbidden

Allowed:

- one-way event export from mod to configured targets
- local JSONL fallback with retention controls
- local sidecar ingest with target sync auth
- sidecar-owned storage, summaries, AX packages, and UI rendering
- explicit user-approved outbound integrations

Forbidden:

- sidecar-to-mod gameplay commands
- auto-clicking/input injection/navigation/combat automation
- secrets in event payloads or diagnostic bundles by default
- remote upload without a configured remote target or explicit user action
- using one key class as a substitute for another
- making sidecar availability a requirement for normal mod behavior

## Change Process

Before adding a new event family, route, target type, or key class:

1. Name the trust zone that owns it.
2. Classify the data and secret material involved.
3. Define the producer pressure valve.
4. Define the consumer pressure valve.
5. Decide whether the target is local-only or remote-capable.
6. Add or update the event/schema/API docs.
7. Add tests for auth, mode gates, redaction, and failure behavior.
8. Dogfood through AX before building UI.
