# Local API Security Model

This model sits under the producer/consumer contract in [docs/22-producer-consumer-security-contract.md](22-producer-consumer-security-contract.md). The contract defines which zone owns a key or data flow; this document defines how localhost routes enforce that ownership.

## Goal

If sidecar core later exposes a localhost HTTP or WebSocket API, it should support:

- a local viewer or desktop shell
- optional overlay clients
- explicit user-initiated export actions

without turning loopback into an implicit trust boundary.

## Threat Model

The local API is not public internet-facing, but it is still exposed to:

- other local user processes
- browser tabs and extensions
- local malware already running in user context
- accidental cross-origin requests from unrelated local pages

The API should therefore assume that localhost access alone is not equivalent to authorization.

## Route Classes

### Class A: Public Read-Only Diagnostics

Examples:

- health
- local viewer status
- redacted event snapshots
- redacted battle summaries already present in local files

Requirements:

- bind to loopback only
- GET only
- no secrets in response bodies
- no mutation side effects

This class may be readable without a capability token when the caller is local and the content is already intentionally viewable.

### Class B: Local Sensitive Read

Examples:

- session inventories with PII-rich detail
- export previews that include coordinates, alliance names, or timing detail
- provider profile metadata

Requirements:

- loopback only
- capability token required
- explicit CORS allowlist
- no browser credential cookies

### Class C: Privileged Mutation / Egress

Examples:

- send battle to Majel
- save or delete provider credentials
- rotate local API token
- create diagnostic bundle
- open outbound WebSocket or sync job

Requirements:

- capability token required
- POST or other non-GET method
- CSRF-resistant request model
- explicit user intent marker when appropriate
- auditable event emission after execution

## Authentication Model

## Local Capability Token

The default auth mechanism should be a launch-scoped local capability token generated by sidecar core.

Properties:

- random, high-entropy secret
- valid only for the current sidecar process lifetime
- not reused as a provider token
- not stored in battle payloads or JSONL
- exposed only to trusted local UI bootstrap paths

Recommended transport:

- `Authorization: Bearer <local-capability-token>` for HTTP
- `Sec-WebSocket-Protocol` or explicit auth message for WebSocket handshake only if the chosen client platform requires it

Avoid query-string tokens except for short-lived local bootstrap situations where headers are impossible.

## Session Bootstrapping

Preferred models, in order:

1. Desktop shell spawns sidecar core and receives the token through process bootstrap.
2. Sidecar core writes the token to a protected local bootstrap file with user-only permissions, then the local UI reads it once.
3. Manual copy/paste during development only.

Avoid storing the local capability token in durable config.

The viewer server must not rely on a hardcoded development sync token. When
`STFC_SIDECAR_SYNC_TOKEN` is not provided, desktop and managed dev launches pass
a launch-scoped generated token to the server. Settings mutation in
`remote_protected` mode requires `STFC_SIDECAR_SETTINGS_TOKEN`; it does not
implicitly reuse the sync ingest token.

## CORS And Browser Rules

If a browser-rendered UI exists, default policy should be strict.

Rules:

- deny all origins by default
- allow only explicit loopback origins used by the sidecar UI
- do not rely on wildcard origins
- do not use cookies for auth
- require the local capability token on privileged routes even for same-origin UI

## Port And Binding Rules

- bind to `127.0.0.1` by default
- prefer a random high port unless a fixed port is operationally necessary
- advertise the chosen port via trusted local bootstrap only
- never bind to `0.0.0.0` by default

## HTTP Recommendations

### Allowed Anonymous Routes

Keep this list intentionally small:

- `/api/health`
- `/api/viewer-config` if fully redacted
- `GET /api/events?detail=summary` and `GET /api/events/stream` while they
  remain read-only summary snapshots or update hints for the local viewer
- `/api/diagnostics/bundle` only while it remains redacted, GET-only, and free
  of raw local paths, tokens, credentials, and raw event payloads

### Token-Protected Routes

Everything else should require the local capability token unless there is a strong reason not to.

Examples:

- `POST /api/events`
- `/api/battles/*`
- `/api/integrations/*`
- `/api/diagnostics/*`
- `/api/credentials/*`

### Developer-Only Routes

Developer Tools are a runtime mode, not a renderer-only preference. Routes under
`/api/dev/*` must return `403 developer_mode_required` unless the viewer server
was started with Developer Tools enabled. Public developer surfaces such as the
battle workbench are also denied by the server in Standard Mode, even when a
caller guesses the URL directly.

### Mutation Rules

- no privileged mutations over GET
- require JSON body with explicit action shape
- return redacted audit metadata

## WebSocket Recommendations

If richer live streaming becomes necessary:

- require capability token during connection setup
- treat each socket as a privilege-bearing session
- support server-side disconnect when token rotates or sidecar restarts
- never stream secret material even over authenticated local sockets

The current Battle Log live update channel uses server-sent events at
`GET /api/events/stream`. It sends only update hints (`ready` and
`events-updated`) and relies on the existing snapshot/detail routes for data.
If the stream begins carrying event payloads, cursors, or sensitive status, move
it behind the local capability token.

## Credential Handling Through The API

The local API should never return raw provider secrets after storage.

Allowed responses:

- profile metadata
- provider name
- profile id
- creation/update timestamps
- scopes
- validation status

Forbidden responses:

- stored bearer token
- refresh token
- cookie jar
- raw auth header

When the UI submits a new provider token:

- send it only over the local capability-protected route
- store it immediately in secret storage
- never echo it back
- emit only redacted integration telemetry

## Bootstrap Credential Policy

For a narrow first integration, a manually configured TOML bearer token can be acceptable if all of the following are true:

- one provider only
- one manually pasted long-lived token
- no refresh-token lifecycle
- no multi-profile switching
- no untrusted local UI process boundary
- no expectation that the token will be excluded from ordinary config backups

Once any of those assumptions stop being true, move to a credential-store-backed flow.

## Audit Model

Privileged API actions should emit redacted `integration.event` entries that capture:

- provider
- profileId
- action
- status
- userInitiated
- failure class if any

They should not emit secret values.

## Recommended Phase Order

### Phase 1

- no local API beyond simple viewer routes
- keep viewer read-only

### Phase 2

- add capability-token auth before adding privileged routes
- define route classes explicitly

### Phase 3

- add provider credential management through sidecar core only
- keep secrets in secret storage

### Phase 4

- add WebSocket or richer UI clients if needed

## Design Summary

The local API should treat loopback as a network location, not as an authorization decision.

Public read-only diagnostics can stay simple. Sensitive reads and all mutations should require a launch-scoped local capability token, while provider credentials remain out-of-band in sidecar secret storage.
